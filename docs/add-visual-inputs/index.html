0. Status, assumptions & open questions

This is a design, not an implementation. The vision components (SigLIP-style tower, mm projection, placeholder expansion, embedding merge) already exist in gemma/gm/nn/gemma4/vision/ and are reused verbatim — their shapes and parameter counts below are measured (tools/rampup/smoke_vision.py). The plumbing that connects them to the diffusion adapter (config, data transforms, mask variants, loss masking, images= threading) does not exist yet; every such item is tagged [DESIGN].
Verified by execution (tools/rampup/measured/vision_*.json): the full 26B_A4B vision tower inventory via jax.eval_shape — 569,550,384 params + 3,244,032 for mm_input_projection → +572,794,416 new params (model total 25,250,986,782 → 25,823,781,198); a tiny-config (D_v=16, 2 layers) image forward through preprocess_image → patchify_and_pad → VisionEncoder → Embedder.encode_vision with every module I/O recorded; variable-aspect-ratio soft-token counts (e.g. 1080×1920 → 576×1056 → 264 soft tokens; 120×200 → 576×1008 → 252); placeholder expansion (<|image|> → span of S_v+4 tokens); and merge_flat_embeddings tolerating per-example variable token counts (excess rows scatter to slot 0, which is then restored).
Assumptions & open questions
  • Checkpoint contains vision weights. Per the feature owner, DIFFUSIONGEMMA_26B_A4B_IT includes the vision tower and projection weights, so GemmaDiffusionCheckpointLoader keeps working unchanged (it raises on any model param missing from the checkpoint, gemma_checkpointer.py:227-242). [UNVERIFIED] locally — if wrong, the fallback is a second init_transform that copies the identically-shaped tower from the AR Gemma4 26B_A4B multimodal checkpoint.
  • Vision tower is trainable. Required by the feature owner. Gemma4's VisionEncoder applies no stop-gradient (unlike Gemma3's SigLiPFromPatches, gm/nn/vision/_vision.py:246,291-292), and stop_gradient_from_denoiser_to_encoder=False (sft_model.py:355-356), so both losses reach the tower: diffusion loss → KV cache → encoder pass → merge → projection → tower, and encoder AR loss → same path. Note the tower + projection are kept float32 (gemma4/_transformer.py:289-301) while the backbone is bf16.
  • remove_mm_logits must be bypassed [DESIGN]. With images≠None and full-sequence logits, Transformer.__call__ strips MM positions using Gemma3 special tokens and a fixed per-image count (gemma4/_transformer.py:318-330; _token_utils.py:348-366) — wrong for gemma4's variable counts and for pre-expanded prompts. The design keeps logits at full length [B, FS, V] and instead masks the whole expanded image span (soft slots and markers) out of the AR loss (see §10, steps 4 and 10).
  • External sliding mask [DESIGN]. With images, the transformer builds its vision-bidirectional sliding mask internally as [B, L, L] from tokens != 0 (gemma4/_transformer.py:460-471) — correct at training prefill (cache = FS) but shape-wrong at inference prefill (cache C > P). The design adds an optional sliding_attention_mask passthrough sized to the cache, built by a new mask_helpers.make_vision_prefill_masks.
  • Batched vision encode [DESIGN]. _encode_vision + PreprocessedVisionInput assume batch-1 inference with static per-image counts (gemma4/_transformer.py:542-573) — a new count combination per batch would trigger a jit recompile. The design runs the tower batched [B, P_p, p_d] → [B, S_max, D_v] + validity mask and scatters per example by the placeholder mask; the excess-rows-to-slot-0 semantics of merge_flat_embeddings make this safe (measured, vision_merge_semantics.json).
  • One image per example (feature-owner scope). Multi-image works by construction (spans concatenate) but grows P; the sliding window (1024) starts to bind once FS > 1024 — with P=384, FS=640 it never does, same as the baseline.
  • Small images are up-scaled. Aspect-preserving resize always targets the full 2520-patch budget (48×48 → 768×768 → 256 soft tokens, measured) — S_v is 250–280 in practice, never "few tokens for a small image".
  • Open recipe question — should the encoder AR loss supervise the prompt at SFT? The shipped SFT configs keep full-sequence AR CE (weight 1.0, prompt included, data.py:131-148) — formally a pretraining-shaped objective. The mainline AR-Gemma SFT stack masks prompts ("0 for prompt, 1 for response", gm/data/_tasks.py:72), and diffusion-LM SFT recipes are unanimous that the prompt stays clean and loss-free (LLaDA eq. 5, arXiv:2502.09992; MMaDA; LLaDA-V; Dream 7B). No published diffusion LM retains an auxiliary AR loss at SFT — though none shares this architecture where the AR pass produces the conditioning KV cache; the closest analogue (BD3-LM's clean-context pass) trains that pass through the diffusion gradient alone. A literature-consistent middle ground is restricting the AR loss to the response/canvas region; encoder_loss_weight=0 is also mechanically safe (nothing else consumes encoder_logits, and the encoder pass still trains via the diffusion loss since stop_gradient_from_denoiser_to_encoder=False). Left as a training-recipe knob — this design changes only the image-span masking.
  • Baseline open questions (time t unused by the network; "A4B" active-param reading; self_cond_prob=0.5 hidden default; unused gemma/diffusion/_sampler.py) are unchanged from docs/model-walkthrough/index.html.

1. The design in one page

DiffusionGemma_26B_A4B (gemma/diffusion/_models.py:21) already inherits a fully multimodal backbone: Gemma4_26B_A4B ships a SigLIP-style VisionEncoder (D_v=1152, 27 layers, ≤280 soft tokens/image) and use_bidirectional_attention='vision' (gemma/gm/nn/gemma4/_gemma4.py:286-295) — but the diffusion configs construct it with the default text_only=True, which deletes the tower at __post_init__ (_gemma4.py:44,50-59). The design flips that switch and threads images through the one place they are needed:

Design principle — vision enters only through the encoder/prefill pass. Images become ≤280 soft tokens spliced into the prompt: the data pipeline expands the <|image|> placeholder into \n\n <start_of_image> [-2]×S_v <end_of_image> \n\n (_token_utils.py:375-434); during the encoder prefill the vision tower embeds the image and merge_flat_embeddings replaces the -2 slots (gemma4/_transformer.py:421-540). From there the image lives in the KV cache like any other prompt token. The denoiser passes, self-conditioning, corruption process, DDIM stepper, canvas machinery and both loss formulas are unchanged — even the decoder attention-mask rule ("attend to valid prompt keys", mask_helpers.py:247-309) already admits image positions, because placeholder/soft tokens are non-PAD.

5.1 Inside sft_encode (encoder prefill)

Expansion of the "sft_encode" node above (sft_model.py:159-217; hd_gemma_network.py:52-107). One standard causal Gemma forward over prompt ++ x0 [B, FS] — the same layer_i/*, embedder and final_norm weights as the denoiser passes, but via Transformer.__call__, so the self_conditioner module is not run. MODIFIED It now also receives images = (patches, positions_xy) and forwards them into Transformer.__call__, where the vision tower runs and the soft tokens are merged into the prompt embeddings — so the prefilled KV cache contains image K/V. It produces both training signals at once: encoder_logits [B, FS, V] for the AR loss, and the prefilled KV cache whose per-example write cursor is then rewound to P + selected_idx·CS so the denoiser reads only prompt + prior canvases as context. In the explorer, this diagram is a drill-down of the sft_encode node (Training forward → sft_encode).

6. Inference sequence (AR diffusion sampling)

Both loops are explicit: the outer autoregressive canvas loop (hackable_diffusion .../ar_diffusion_sampler.py:244-332) and the inner denoising loop (.../sampling.py:190-281). With the production config NC = max_num_canvases = 1, the outer loop body runs once. The early-stopping evaluator variant wraps the inner loop in a while that exits when mean token entropy ≤ 0.05 (.../diffusion_early_stopping.py:117-134). MODIFIED Vision touches this flow in exactly one place — the prompt prefill inside init_ar_state (purple region): the evaluator adds the patch tensors to the conditioning dict, and the prefill's encoder forward embeds + merges them so the image K/V sit in the cache before the first canvas. Both loops are untouched.

10. Life of a Batch (training)

Steps tagged NEW / MODIFIED are [DESIGN]; the rest is the verified baseline behavior.

  1. Read + format.   2. Prompt prefill.     3. Train vs inference network call: same backbone method; differences are only in orchestration — cache prefilled from prompt ++ x0 (train) vs prompt-only then appended per canvas (inference); queries = all canvases [B,TC,C] (train) vs one canvas [B,CS,C] (inference); self-conditioning two-pass w.p. 0.5 (train) vs every step from the previous iteration (inference). In both regimes the vision tower runs exactly once per sequence, inside the prompt/encoder prefill.
